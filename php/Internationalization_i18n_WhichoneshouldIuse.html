Internationalization (i18n): Which one should I use? __(), _e(), _n()

You should use them based on how you want to handle the output (echo vs. return) and whether the text depends on a count (singular vs. plural). They are not interchangeable; each has a very specific architectural role.

Here is the breakdown of when to prefer each one.

1. Prefer __() when you need to RETURN a string

The double-underscore function translates the text and returns it as a string variable. It does not print anything to the screen automatically.

Best Used For:

  • Passing text inside other PHP functions (like configuration arrays, shortcodes, or hooks).
  • Storing text inside variables for later use.
  • Passing text into escaping functions (highly recommended).

PHP

// Storing in a variable to use later

$page_title = __( 'Bookstore Dashboard', 'text-domain' );

// Passing text into a WordPress button layout configuration

add_submenu_page( 'tools.php', __( 'Book Admin', 'text-domain' ), ... )

;

2. Prefer _e() when you want to ECHO text directly

The _e() function stands for "Translate and Echo". It translates the text and immediately prints it directly to the HTML output stream.

Best Used For:

  • Raw HTML template files where you just need to display static text inside tags (like headings, paragraphs, or labels).

PHP

<h1

><?php echo __( 'Welcome', 'text-domain' ); ?></h1>

<h1><?php _e( 'Welcome', 'text-domain' ); ?></h1>

3. Prefer _n() when text changes based on a number (Plurals)

The _n() function handles pluralization conversions. It accepts a singular sentence, a plural sentence, and an integer count. WordPress looks at the number and automatically decides which version of the sentence to display.

Best Used For:

  • Dynamic UI status elements, cart tallies, or result readouts where showing something ugly like "1 books" or "0 book" would look unprofessional.

PHP

$book_count = 3; // Dynamically fetched from database

echo _n(

    'You have 1 book.',          // Singular choice (if $book_count == 1)

    'You have multiple books.',  // Plural choice   (if $book_count != 1)

    $book_count,                 // The control variable

    'text-domain'                // Your text domain

);

⚠️ The Security Catch: The "Escaped" Pro-Tips

In modern WordPress development, it is a strict requirement to escape strings while translating them so malicious database translations can't hijack your HTML structure.

Because of this, experienced developers rarely use raw __() or _e(). Instead, they prefer using these built-in translation + escaping combination wrappers:

  • Instead of echo esc_html( __( 'Text', 'slug' ) ); ➡️ Prefer esc_html__( 'Text', 'slug' )
  • Instead of esc_attr( _e( 'Text', 'slug' ) ); ➡️ Prefer esc_attr_e( 'Text', 'slug' )

Is internationalization (i18n) compulsory for personal or single client projects?

internationalization (i18n) is absolutely not compulsory if you are writing a theme or plugin strictly for your own personal use or for a single local client. If you know the site will only ever be in English, writing raw strings like echo 'Add Book'; will work perfectly fine, and your code will execute without any issues.

However, even for personal projects, many experienced WordPress developers still choose to internationalize their code. Here is a quick breakdown of why it’s not required for you, but why it might still be a good habit to keep.

Why You Can Safely Skip It For Personal Use

  • No Repository Rules: The strict requirement for text domains and translation functions (__(), _e(), esc_html__()) is enforced by the automated WordPress.org systems. Since you aren't uploading your code to the public plugin directory, no one will reject it.
  • Slightly Faster Writing: You don't have to constantly wrap your text strings in functions or append your text domain name at the end of every sentence.
  • Cleaner Looking HTML: Writing <h1>Bookstore</h1> is visually cleaner than reading <h1><?php echo esc_html__( 'Bookstore', 'my-plugin-slug' ); ?></h1>.

3 Reasons to Consider Doing It Anyway

If you have the time, getting into the habit of internationalizing your personal code offers a few hidden benefits:

1. It forces proper HTML escaping

In WordPress, the most common way to translate text is to use translation + escaping combos like esc_html__() or esc_attr__(). When you get used to writing translation strings, you naturally end up writing highly secure, escaped code by default:

PHP

// If you skip i18n, you might forget to escape:

echo '<input placeholder="' . $placeholder . '">';

// If you use i18n, you naturally use the safe escaping wrapper:

echo '<input placeholder="' . esc_attr__( 'Search Books...', 'text-domain' ) . '">';

2. Future Client Flexibility

A personal project today might turn into a premium plugin or a framework you sell to a client tomorrow. If a client suddenly requests that their site support a second language (like Spanish or French) six months from now, finding and wrapping hundreds of raw text strings across dozens of files is an incredibly tedious headache.

3. Code Reusability

As you build more plugins, you will find yourself copying and pasting your favorite utility functions from old projects into new ones. If those snippets are already internationalized, they are instantly ready for any environment you drop them into.

The Verdict

For your current local layout testing and personal development: Save your time and skip it! Write your text as plain, simple strings.

Just keep in mind that if you ever decide to distribute your plugin publicly or submit it to the official WordPress repository, adding a text domain will be the very first thing you have to fix.
